## Phase 1: Event Management + IaC & CI/CD Baseline

**Step 1: Creating IaC + deployment**: Bicep templates created, resources deployed via azure CLI locally and everything works. Added ADRs for cosmosdb, hosting plans and managed identity auth.

**Step 2: CI/CD pipeline**: GitHub Actions workflow configured to build, test, and deploy on every push to `main`. Infrastructure is kept in sync via an idempotent Bicep deployment on each run, supporting dev, stage, and prod environments each with their own region and secrets. An Azure AD app registration with a system-assigned service principal was created, granted Owner at subscription scope (required for RBAC role assignments in Bicep), and wired to GitHub Actions via OIDC federated credentials — one per environment — so no client secrets are stored anywhere.

**Step 3: Functions project bootstrap**: Scaffolded the isolated worker entry point (`Program.cs`, `host.json`) and added the `Microsoft.Azure.Functions.Worker` and `Microsoft.Azure.Functions.Worker.Sdk` packages, enabling `dotnet publish` to generate the `.azurefunctions` metadata directory required by Kudu for zip deployment.

**Step 4: Local and cloud dev environment verification**: Set up `docker-compose.yml` with Azurite (ports 10000-10002) and the Cosmos Emulator (port 8081, with a healthcheck). Added `requests/health.http` for one-click health checks via REST Client. `Program.cs` calls `EnsureCreatedAsync()` at startup (wrapped in a try-catch so a transient failure logs a warning instead of crashing the worker). Two `local.settings` templates exist: the default `local.settings.json` targets emulators; `local.settings.azurecli.json` targets live Azure using `AzureCliCredential` with identity-based storage (`AzureWebJobsStorage__accountName`). Both files are gitignored. The reliable way to run locally is `dotnet run --project src/TicketFlow.Functions` — the SDK-bundled func CLI (v4.6.x) handles net10.0 correctly. The IntelliJ/Rider-bundled func (v4.8.0) has a known issue on net10.0 where `local.settings.json` values are not passed to the isolated worker; workaround is to set the required env vars directly in the Rider run configuration. A further Rider gotcha: `AzureCliCredential` shells out to `az account get-access-token`, but Rider launches child processes with a sanitised PATH — add `/usr/bin:/usr/local/bin` to the PATH env var in the run configuration so the `az` binary can be found.
