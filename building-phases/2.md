## Phase 2: Ticket Purchase — Basic Orchestration

**Acceptance Criteria:** As an attendee I can submit a purchase request. The system reserves a seat, runs a simulated payment (we can set status of payment from request to simplify), and marks the order as confirmed or failed — all as a Durable Functions orchestration.

**Azure Resources to Add:**

- Cosmos DB `orders` container
- Durable Functions extension _(no new Azure resource, just SDK + storage used by runtime)_

**What you build:**

- `POST /orders` — triggers the orchestrator
- Orchestrator: reserve seat → payment (simulated) → update order status
- Optimistic concurrency on `availableSeats` using `_etag`
- `GET /orders/{orderId}` — poll orchestration status
